## 🐴 프롬프트 엔지니어링은 끝났습니다: 이제 '하네스(Harness)'의 시대입니다
최근 오픈AI의 엔지니어 3명이 코드를 한 줄도 직접 쓰지 않고 5개월 만에 소프트웨어 제품을 배포한 사례가 화제가 되었습니다. 이들이 한 일은 코딩이 아니라, AI가 안전하게 코드를 만들 수 있는 환경을 설계한 것이며, 이를 **'하네스 엔지니어링(Harness Engineering)'**이라고 부릅니다.

1. 기존 엔지니어링의 한계와 하네스의 필요성
    AI를 활용하는 방법론은 크게 4가지 축(프롬프트, 컨텍스트, 에이전틱, 하네스)으로 나뉩니다.

   - 프롬프트 및 컨텍스트 엔지니어링의 한계: 프롬프트를 정교하게 쓰고(프롬프트 엔지니어링) 필요한 정보(컨텍스트 엔지니어링)를 아무리 잘 줘도, AI가 규칙을 어기거나 엉뚱한 짓(예: 신용카드 번호 노출, DB 스키마 임의 변경)을 하는 것을 막을 수는 없습니다. 이는 정보 부족이 아닌 규칙과 통제(울타리)의 문제입니다.

   - 말과 마구(Harness)의 비유:

      - AI 에이전트 = 강력한 말: 무거운 짐을 끌고 밭을 갈 수 있지만, 통제하지 않으면 밭을 망치고 도망갑니다.

      - 에이전틱 엔지니어링 = 말을 훈련시키는 기술: 추론하고 도구를 쓰도록 AI 자체를 강화합니다.

      - 하네스 엔지니어링 = 마구(가죽끈, 고삐, 수레)를 만드는 기술: 말이 밭을 제대로 갈 수 있도록 방향과 한계, 가드레일을 설정하는 기술입니다.

💡 하네스의 핵심 철학: AI가 실수했을 때 "다음엔 잘해"라고 프롬프트를 고치는 것이 아니라, 그 실패가 구조적으로 반복 불가능하도록 시스템(울타리)을 고치고 더 튼튼하게 만드는 것입니다.

1. 하네스 엔지니어링의 4가지 기둥 (Pillars)
    하네스 환경은 다음 4가지 요소로 구성됩니다.

    1. 기계가 읽는 컨텍스트 파일 (Machine-readable Context): .cursorrules, agents.md와 같이 AI가 작업 전 가장 먼저 읽는 런타임 설정 파일입니다. 특정 라이브러리 금지, API 패턴 등 AI의 행동 제약을 설정합니다.

    2. 시스템에 의한 규칙 강제 (Automated Enforcement): 문서에 적어두는 것을 넘어 시스템이 막도록 합니다. 린터(Linter), 구조적 테스트, 프리커밋 훅(Pre-commit hook) 등을 통해 AI가 규칙을 어긴 코드를 작성하면 에러를 띄워 자동으로 수정을 유도합니다.

    3. 물리적 도구 및 접근 제한 (Boundary & Constraints): 애초에 위험한 작업을 하지 못하도록 권한을 물리적으로 차단합니다. (예: 중요 폴더는 읽기 전용으로 설정, DB는 SELECT만 허용 등).

    4. 가비지 컬렉션 (자동 청소 시스템): AI는 기존 코드의 나쁜 패턴을 복제하는 경향이 있습니다. 따라서 사용하지 않는 데드 코드 제거, 리팩토링, 안티패턴을 주기적으로 감지하고 청소하는 시스템이 필요합니다.

2. 하네스 시스템의 4가지 작동 구조
    실제 시스템은 다음 4개의 부품이 맞물려 돌아갑니다.

    - 라우터 (Router/Classifier): 사용자의 요청이 모호한지 실제 작업인지 판별하여, 엉뚱한 방향으로 가는 것을 출발 전에 막습니다.

    - 컨텍스트 매니저 (Context Manager): AI에게 전체 코드를 보여주지 않고, 지금 당장 필요한 파일과 규칙(섹션)만 똑똑하게 제공하여 시야를 좁혀줍니다 (가림막 역할).

    - 실행 루프 (Execution Loop - 하네스의 심장): AI가 코드를 작성 ➡️ 자동 테스트 ➡️ 실패 시 에러 메시지를 AI에게 다시 전달하여 스스로 수정. (인간 개입 불필요)

    - 워커 격리 (Worker Isolation): 코드를 작성하는 AI와 검토(리뷰)하는 AI의 역할을 완전히 분리하여 품질을 높입니다.

3. 결론: AI 시대, 우리의 역할 변화
    "인간은 조종하고 에이전트는 실행한다."

    AI의 발전으로 코딩 능력이 불필요해지는 것이 아닙니다. 오히려 개발자든 비개발자든 요구되는 능력이 **'상향 이동'**하고 있습니다.
    단순히 코드 한 줄을 쓰는 선수(Player)의 역할에서, AI가 실수 없이 일할 수 있는 거대한 시스템과 전술을 설계하고, 최종 책임을 지는 **감독(Manager/Architect)**의 역할로 진화해야 합니다.
