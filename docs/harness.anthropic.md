## 🏗️ Harness Design for Long-Running Application Development

> **저자**: Prithvi Rajasekaran (Anthropic Labs) | **발행일**: 2026년 3월 24일

### 핵심 문제 정의

저자가 해결하려던 두 가지 연결된 문제:
1. Claude가 고품질 프론트엔드 디자인을 생성하도록 개선
2. 인간 개입 없이 완전한 애플리케이션 구축

이전 방식(프롬프트 엔지니어링, 기본 harness)은 한계에 도달함.

### GAN 영감 구조 (Generator + Evaluator)

생성적 적대 신경망(GAN)에서 영감을 받아 **생성자(Generator)** 와 **평가자(Evaluator)** 멀티 에이전트 구조를 도입합니다:
- 주관적 판단("이 디자인이 좋은가?")을 구체적 기준으로 변환
- 작업 수행 에이전트와 판단 에이전트를 분리

> "Separating the agent doing the work from the agent judging it proves to be a strong lever"

---

### ⚠️ Naive 구현이 실패하는 이유

#### 1. Context 불안감 (Context Anxiety)

모델이 컨텍스트 창이 꽉 찰 때마다 작업의 일관성을 잃습니다.

- **축약(Compaction)**: 연속성은 유지되지만 불안감이 지속됨
- **초기화(Full Reset)**: 깨끗한 상태를 제공 → **권장 방식**

#### 2. 자기 평가의 편향

"당신의 작업을 평가하세요"라고 하면 모델은 자신의 결과를 자신감 있게 칭찬하고 객관적 품질 판단이 부족해집니다. 이를 해결하기 위해 독립적인 평가자 에이전트를 도입합니다.

---

### 🎨 프론트엔드 디자인: 주관성을 등급화하기

#### 4가지 평가 기준

| 기준 | 설명 |
|------|------|
| **Design Quality** | 색상, 타이포그래피, 레이아웃, 이미지의 조화로운 분위기 창조 |
| **Originality** | 맞춤형 결정 증거; 템플릿/기본값 의존 금지; "AI 슬로프" 패턴 제외 |
| **Craft** | 타이포그래피 계층, 간격 일관성, 색상 조화, 명도비 |
| **Functionality** | 미학과 독립적 사용성: 인터페이스 이해, 주요 작업 발견 용이성 |

**가중치**: Claude가 기본적으로 잘 수행하는 Craft와 Functionality보다 **Design Quality와 Originality**를 더 중점적으로 평가합니다.

#### 반복 프로세스

```
Generator (HTML/CSS/JS 생성)
    ↓
Evaluator (Playwright MCP로 실제 페이지 상호작용 + 스크린샷)
    ↓
상세 피드백 전달
    ↓
다음 반복 (총 5~15회 반복)
```

#### 사례: 네덜란드 미술관 웹사이트

- **9번째 반복**: 어두운 테마 랜딩페이지 (기대치 충족)
- **10번째 반복**: 완전 재구상 → 3D 공간 경험
  - CSS 원근법으로 바닥 렌더링
  - 벽에 자유형식으로 배치된 미술작품
  - 스크롤 대신 문을 통한 갤러리 네비게이션

핵심 발견: 프롬프트 표현("박물관 품질의 디자인")이 생성 결과의 성격을 직접 형성합니다.

---

### 🛠️ 풀스택 코딩으로 확장: 3-에이전트 아키텍처

#### 1. Planner (계획자)

1~4문장의 간단한 프롬프트를 받아 전체 제품 사양으로 확장합니다. 기술 구현보다 제품 컨텍스트 중심으로 야심찬 범위를 설정하며 AI 기능을 자동 통합합니다.

예시: "2D 레트로 게임 메이커 만들기"
→ RetroForge 제품 스펙 (16개 기능, 10개 스프린트) 자동 생성

#### 2. Generator (생성자)

- React + Vite + FastAPI + SQLite/PostgreSQL 스택 사용
- 스프린트 방식으로 구현 (한 번에 하나씩)
- 각 스프린트마다 자체 평가
- Git 버전 관리

#### 3. Evaluator (평가자)

- Playwright MCP로 실제 애플리케이션 테스트
- UI 기능, API 엔드포인트, DB 상태 검증
- **스프린트 계약** 협상: 코드 작성 전 "완료" 기준 정의 (Generator 제안 → Evaluator 검토 → 합의)

### 📊 실험 결과: 레트로 게임 메이커 (단일 에이전트 vs. 전체 Harness)

| 항목 | 단일 에이전트 | 전체 Harness |
|------|--------------|-------------|
| **소요 시간** | 20분 | 6시간 |
| **비용** | $9 | $200 |
| **출력 품질** | 낮음 | 높음 |

단일 에이전트의 문제점: 레이아웃 공간 낭비, 불명확한 워크플로우, **게임 핵심 기능 파손** (엔티티 정의와 게임 런타임 간 연결 끊김)

전체 Harness의 장점: 세련된 UI, 일관된 시각적 아이덴티티, 직관적 워크플로우, **실제 게임 플레이 가능**

### 🔄 Harness 단순화 (Opus 4.6 이후)

"최소 솔루션을 찾아 필요할 때만 복잡도를 증가시킨다"는 원칙 아래 구성 요소를 지속적으로 검토합니다.

- **Opus 4.5** → 컨텍스트 초기화 + 스프린트 구조 필요
- **Opus 4.6** → 연속 세션 + 자동 압축으로 스프린트 구조 불필요
- 평가자는 작업이 모델 기본 능력 범위 내에 있으면 생략 가능

### 🎵 DAW(디지털 오디오 워크스테이션) 사례

개선된 Harness의 실제 비용 및 시간 구조:

```
Planner:       4.7분,  $0.46
Build Round 1: 2h 7m,  $71.08
QA Round 1:    8.8분,  $3.24
Build Round 2: 1h 2m,  $36.89
QA Round 2:    6.8분,  $3.09
Build Round 3: 10.9분, $5.88
QA Round 3:    9.6분,  $4.06
────────────────────────────
합계: 3h 50m, $124.70
```

QA를 통해 발견된 문제들(오디오 녹음 미작동, 클립 리사이징 미구현 등)이 각 라운드에서 수정되어 최종적으로 기능적인 음악 제작 프로그램이 완성되었습니다.

### 💡 주요 교훈

모델이 개선될수록 Harness의 복잡도는 줄일 수 있지만, 새로운 능력을 위한 **새로운 Harness 조합**이 생겨납니다. AI 엔지니어의 역할은 이 새로운 조합을 지속적으로 찾는 것입니다.

세 가지 핵심 원칙:
1. 모델과 함께 작업하며 실제 문제를 추적한다
2. 복잡한 작업은 특화된 에이전트를 활용한다
3. 새 모델 출시 시 Harness를 재검토하고 제거 가능한 부분을 제거한다
